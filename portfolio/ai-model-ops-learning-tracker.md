# AI 模型运营求职学习 Tracker

## 用途

这个文档用来记录 ncubook 求职项目的学习规划、每日产出和掌握状态。

目标不是把所有大模型技术都学成研究员水平，而是在一周内达到：

1. 能看懂岗位 JD 中的核心词。
2. 能把 ncubook 的功能映射到岗位能力。
3. 能解释自己做了什么、没做什么、为什么这样做。
4. 能通过项目产出证明 AI 产品运营、模型运营、Agent 产品和模型策略产品相关能力。

## 学习方法

每个主题都按同一个节奏推进：

1. 概念白话解释：先听懂这个词到底解决什么问题。
2. 放回 ncubook：判断项目里哪里体现了这个能力。
3. 你来复述：用自己的话讲一遍，我负责纠偏。
4. 小任务落地：改一个 prompt、补一个 case、写一段分析或看一个指标。
5. 形成求职材料：沉淀到 README、复盘、简历或面试稿里。

## 一周学习与产出计划

### Day 1：大模型训练链路

目标：把岗位 JD 里的 Pretrain、SFT、RLHF、DPO、推理优化、RAG、Prompt、Eval 放到同一张地图里。

要学会：

1. Pretrain 是让模型获得通用语言和知识能力。
2. SFT 是用人工示范数据教模型按任务格式回答。
3. RLHF / DPO 是用偏好数据让模型更符合人类期待。
4. 推理优化是让模型在调用时更快、更稳、更便宜。
5. RAG 是在推理时给模型补外部知识。
6. Prompt 是控制模型当下如何回答。
7. Eval 是判断模型和系统效果是否真的变好。

ncubook 对应关系：

| 链路 | ncubook 是否做了 | 怎么讲 |
| --- | --- | --- |
| Pretrain | 没做 | 不宣称训练基础模型 |
| SFT | 没做 | 不宣称微调模型 |
| RLHF / DPO | 没做 | 用户反馈用于产品迭代，不等同于真正 RLHF |
| RAG | 做了 | 用校园文档向量检索作为回答依据 |
| Prompt | 做了 | 约束来源展示、低命中处理和高风险边界 |
| Eval | 做了 | 用测试集、badcase 和真实反馈评估效果 |
| 数据反馈闭环 | 做了 | query log -> feedback -> gap -> 内容更新 -> 再评测 |

当天产出：

1. 一张“模型训练链路 vs ncubook 能力边界”说明。
2. 一段面试表达：为什么这个项目不是模型训练项目，但和模型运营岗位相关。

### Day 2：模型能力边界与问题定位

目标：学会看到一个坏回答时，判断问题出在哪里。

要学会：

1. 检索没找到资料：优先查知识库和切片。
2. 找到了资料但答错：优先查 Prompt、回答生成和 Trust Boundary。
3. 格式错或没给来源：优先查 Prompt 和接口输出约束。
4. 延迟高或失败：优先查 API、服务器和模型服务稳定性。
5. 真实资料也无法支撑回答：应该进入知识缺口，不该硬答。

当天产出：

1. badcase 归因表。
2. “问题现象 -> 可能原因 -> 排查路径 -> 解决手段”文档片段。

### Day 3：训练数据策略与样本构造

目标：理解模型策略产品 JD 里的“数据分布分析、样本构造、数据质量评估”。

要学会：

1. eval case 不应只靠自己脑补，正式版本应来自真实用户 query、负反馈和知识缺口。
2. 样本要覆盖高频问题、风险问题、低命中问题、格式问题和边界问题。
3. 每条样本要有问题、场景、期望行为、风险等级、mustMention、来源要求。
4. 数据质量不只是数量，而是代表性、准确性、可判定性和可复现性。

当天产出：

1. eval 候选池筛选规则。
2. 真实用户测试后的样本采集模板。

### Day 4：Prompt Engineering 与 Trust Boundary

目标：能把 Prompt 优化讲成“业务规则和模型边界设计”，而不是玄学调参。

要学会：

1. Prompt 需要明确角色、任务、输出格式、引用来源、拒答条件。
2. 高风险问题必须提示核实官方信息。
3. 弱命中 / 无命中必须说明资料不足并记录知识缺口。
4. 不能把“更像人话”放在“更可靠”前面。
5. Prompt 修改必须能对应 badcase。

当天产出：

1. Prompt 迭代记录。
2. 优化前后 eval / badcase 对比说明。

### Day 5：Agent 协作与信息验证

目标：理解多 Agent 不是炫技，而是任务拆解和信息验证机制。

要学会：

1. Agent 本质上是带目标、工具、状态和决策流程的模型调用系统。
2. 多 Agent 适合信息验证、任务拆解、交叉检查、复杂流程编排。
3. ncubook 可以先做轻量 Agent 流程，不必硬堆多个模型角色。
4. 搜索场景中的核心挑战是信息验证、主观判断和多轮决策。

当天产出：

1. ncubook 轻量 Agent 流程图。
2. 判断是否需要 source verifier / answer generator / risk checker 的产品说明。

### Day 6：运营指标与平台化

目标：把项目从“能回答”升级为“能被运营”。

要学会：

1. query volume 看使用量。
2. weak / no retrieval rate 看知识库覆盖和检索质量。
3. negative feedback rate 看回答是否满足用户。
4. avg latency 看体验和服务稳定性。
5. open knowledge gaps 看待处理运营任务。
6. gap resolution 看运营闭环有没有推进。

当天产出：

1. /ops 指标解释。
2. README 中的 AI 运营指标说明。
3. 面试中解释“为什么这不是普通聊天机器人”的话术。

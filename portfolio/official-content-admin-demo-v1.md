# 官网信息运营中台 Demo v1

日期：2026-04-25  
项目：此间 / 南昌大学生存手册  
定位：校园信息运营中台，不是普通官网爬虫。

## 解决什么问题

学生真正关心的不是“学校又发了什么新闻”，而是：

- 哪些信息会影响办事、资格、时间、材料、费用、入口和政策。
- 哪些手册页面需要更新。
- 哪些 AI 回答失败来自知识库缺口。
- 运营者如何在不自动发布的前提下，让 AI 帮忙写出可审核初稿。

所以第一版中台只处理高价值职能部门来源：教务、学工、资助、网络信息、后勤和学校主站学生服务通知。学院新闻、会议、领导调研、活动报道和宣传稿默认排除。

## 为什么先筛选再让 AI 写

如果直接让 AI 读官网并写文章，会有三个问题：

- 噪音大：大量会议新闻和宣传稿会进入队列。
- 成本高：每条内容都调用模型，会浪费 token 和审核时间。
- 风险高：涉及资格、名额、费用、处分、奖助学金、转专业时，AI 不能直接给定论。

当前设计是三层筛选：

1. 来源层：只监听白名单栏目和职能部门来源。
2. 规则层：命中办事、流程、申请、名单、公示、缴费、报修、开放时间、材料、系统入口等关键词才进入候选；会议、调研、走访、活动报道、新闻、领导动态等默认排除。
3. AI 判定层：只对规则命中的候选做二次判断，输出是否入队、原因、风险级别、建议页面和建议动作。

## Trust Boundary

AI 可以做：

- 判断是否值得进入内容运营队列。
- 总结官网原文，生成可编辑 Markdown 初稿。
- 提醒适用对象、时间范围、办理路径和待核实字段。
- 建议更新到哪个手册页面。

AI 不能做：

- 自动发布到学生端。
- 替代官方通知给确定结论。
- 自动修改正式手册文档。
- 对资格、名额、费用、处分、奖助学金、转专业等高风险信息下绝对判断。

所有候选内容必须经过人工编辑、保存、审核通过后，才能导出 Markdown 修改稿。

## 一条完整 Demo 流程

演示路径：

1. 打开 `/admin`，输入 Admin Token 加载中台。
2. 在“信息雷达”查看来源数量、今日抓取、候选数量、高风险任务。
3. 点击“手动抓取官网”，系统抓取白名单来源并保存快照。
4. 进入“候选队列”，查看筛选原因、风险级别、建议动作和来源链接。
5. 选择一条转专业、奖助学金、后勤服务或校园网相关候选，点击“生成 AI 初稿”。
6. 在任务详情中编辑 Markdown，补充人工核实备注。
7. 点击“审核通过”，生成最终 Markdown 修改稿、来源清单和修改理由。
8. 切到“质量与反馈”，查看小家园问答的负反馈、弱命中和知识缺口，说明这些问题可以回流成内容补全任务。

## 本地验收命令

```bash
cd ncubook-api
npm run check:admin
npm run demo:admin
npm run test:source-filter
npm run build
```

如果 `check:admin` 提示找不到 `official_sources`、`official_snapshots`、`official_candidate_events`、`content_drafts` 或 `content_audit_logs`，需要先在 Supabase SQL Editor 运行：

```text
ncubook-api/supabase-official-content-admin-migration.sql
```

`demo:admin` 会真实写入 Supabase：抓取官网来源、生成候选事件、生成 AI 初稿、模拟人工编辑并审核通过。

## 筛选器验收样例

已覆盖的测试场景：

- 学院会议新闻：`ignored`
- 领导调研 / 走访：`ignored`
- 普通活动通知：不能只因为“通知”入队
- 转专业通知：`candidate` + `high` + `docs/academics/major-change.md`
- 奖助学金通知：`candidate` + `high` + `docs/career/awards.md`
- 后勤服务时间变化：关联 `docs/campus-life/repair.md`
- 校园网服务变化：关联 `docs/onboarding/network.md`
- 一码通服务变化：关联 `docs/onboarding/campus-card.md`

运行：

```bash
cd ncubook-api
npm run test:source-filter
```

## 和原 `/ops` 的关系

原 `/ops` 不再作为一个孤立后台故事，而是并入 `/admin` 的“质量与反馈”模块：

- Query Log：学生问了什么，检索命中如何。
- Knowledge Gap：哪些问题没有资料或资料弱。
- Negative Feedback：哪些回答或页面被点踩。
- Gap Status：`open`、`triaged`、`in_progress`、`resolved`、`ignored`。
- 内容回流：某个知识缺口是否已经生成官网抓取任务、AI 草稿或文档修改稿。

这让作品集叙事从“我做了一个问答机器人”升级为“我做了一个可运营、可评估、可回流的校园知识 Agent 系统”。

## 面试讲法

可以这样讲：

> 我没有把官网抓取做成全量爬虫，因为全量爬取会制造大量运营噪音。我的设计是校园信息运营中台：先用来源白名单和规则筛选挡掉新闻、会议、宣传稿，再把真正影响学生办事和权益的信息交给 AI 生成可审核草稿。AI 只负责筛选解释、摘要和初稿，不负责自动发布。最终由运营者审核后导出 Markdown 合并进手册。同时，小家园问答里的负反馈和知识缺口会进入质量模块，反向驱动下一轮内容补全。

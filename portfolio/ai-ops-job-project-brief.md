# ncubook AI 运营求职项目 Brief

## 项目目标

把 `/Users/water/Desktop/求职项目/ncubook` 改造成面向 AI 产品运营、模型运营、Agent 产品实习的求职项目。

这个项目的目标不是单纯继续写功能，而是把 ncubook 打造成一个能写进简历、能在面试中讲清楚的 AI 运营项目。

## 项目现状

ncubook 是一个南昌大学校园信息 / 生存手册项目，不只是内容站。它目前包含：

1. Docusaurus 文档站，已有校园学习、生活、发展等内容。
2. Next.js API 服务 `ncubook-api`。
3. RAG 问答：Gemini embedding + Supabase 向量检索 + DeepSeek 生成回答。
4. Agent Query Log：记录问题、当前页面、检索状态、`source_count`、`max_similarity`、`latency`。
5. Knowledge Gap：当检索 `weak` / `none` 时自动形成知识缺口。
6. 用户反馈：AI 回答 / 页面反馈可写入 Supabase，负反馈同步到飞书。
7. 官网信息运营中台：抓取官方来源，规则筛选 + AI 二次判断，生成候选内容和审核草稿。
8. Eval：已有 `campus-qa-v1.json` 测试集和 `run-eval.ts` 批量评测脚本。

## 目标岗位

1. AI 产品运营实习生
2. 模型运营实习生
3. Agent 产品实习生
4. 内容质量 / 数据服务平台运营
5. RAG / 知识库 / AI 搜索相关产品运营

## 项目要证明的能力

1. AI 产品运营能力：能理解真实用户问题，并把反馈转成产品 / 内容迭代。
2. 模型运营能力：能通过 Query Log、检索状态、badcase、Eval 判断模型效果。
3. Prompt / Agent 策略能力：能设计回答规则、风险边界、来源展示和弱命中处理。
4. 知识库运营能力：能维护内容来源、识别内容缺口、处理过期 / 高风险信息。
5. 数据分析能力：能看查询量、弱命中率、负反馈率、平均延迟、知识缺口数量。
6. 产品表达能力：能写 PRD、项目复盘、简历项目描述，并面向 JD 讲清楚价值。

## 边做边学习的具体技能

1. RAG 基础：embedding、向量检索、相似度、top-k、检索命中状态、幻觉控制。
2. LLM Eval：测试集设计、`mustMention`、风险问题、通过率、badcase 分析。
3. Prompt Engineering：结构化输出、来源引用、低命中处理、风险边界、继续追问设计。
4. 模型运营指标：query volume、weak/no retrieval rate、negative feedback rate、latency、gap resolution。
5. SQL / Supabase 基础：查看日志表、反馈表、知识缺口表，能做简单统计。
6. 产品运营闭环：用户提问 -> AI 回答 -> 反馈 -> 缺口 -> 内容更新 -> 重新入库 -> 再评测。
7. 求职表达：把技术功能翻译成岗位能力和业务价值。

## 项目迭代方向

### P0：Eval / Badcase 闭环

目标：证明这个 AI 产品可评估、可定位问题、可迭代。

任务：

1. 把 eval cases 从 20 条扩展到 40-50 条。
2. 覆盖低风险、中风险、高风险校园问题。
3. 跑一次 baseline eval，记录通过率。
4. 输出一份 badcase 分析报告。
5. 失败原因分为：检索失败、知识库缺失、Prompt 约束不足、风险边界不清。
6. 根据 badcase 优化 prompt / 知识库 / 检索策略，再跑一次 eval，对比前后变化。

### P1：运营看板

目标：让面试官看到这不是聊天机器人，而是可运营、可评估、可迭代的 AI 产品。

核心指标：

1. `totalQueries`
2. `weakOrNoRetrieval`
3. `openKnowledgeGaps`
4. `feedbackCount`
5. `notHelpfulFeedback`
6. `avgLatencyMs`
7. `recentQueries`
8. `recentGaps`

### P1：知识缺口处理流程

目标：证明能把用户问题转成运营任务。

状态流转：

```text
open -> triaged -> in_progress -> resolved / ignored
```

每个缺口最好记录：

1. 问题
2. 来源路径
3. 触发原因
4. 出现次数
5. 处理说明
6. 关联文档

### P2：Prompt / Trust Boundary 优化

目标：证明懂模型回答边界，而不是只会调用模型。

规则：

1. 高风险问题必须提示核实，比如转专业、奖学金、重修、成绩、费用、时间、资格。
2. 弱命中 / 无命中不能编造，要提示“信息需要进一步核实，并已记录为知识缺口”。
3. 回答必须带信息来源。
4. 输出 1 份 Prompt 迭代记录，说明原始问题、修改点、效果变化。

### P2：真实用户测试

目标：让项目不是自嗨 demo。

任务：

1. 找 10-30 个同学试用。
2. 收集真实 query 和反馈。
3. 记录用户最常问的问题、负反馈、未覆盖内容。
4. 用这些数据补充项目复盘。

### P3：求职材料沉淀

最终需要产出：

1. 一份项目 README：说明项目定位、功能、架构、运行方式、求职亮点。
2. 一份项目复盘文档：问题背景、用户、方案、架构、指标、eval、badcase、迭代。
3. 一组截图：学生端问答、信息来源、反馈入口、运营看板、知识缺口、eval 结果。
4. 一版简历项目描述。
5. 一段 2 分钟面试讲述稿。

## 不要做的事

1. 不要为了炫技硬做多 Agent。
2. 不要宣称自己做了模型训练、SFT、RLHF，除非真的做了。
3. 不要把重点放在 UI 美化。
4. 不要做和真实用户无关的套壳 AI demo。
5. 不要把项目包装成算法项目，它应该是 AI 产品运营 / 模型运营 / RAG 知识运营项目。
